When NOT to use a language model.

Four production problems where teams reached for an LLM and ended up with worse outcomes than the classical alternative.

SD
[Principal Engineer]
Principal Engineer · Alpine Quantum
May 12, 2026 11 min read

Every engineering team we work with has an LLM-shaped hammer. Some have legitimate reasons for it — production agentic systems, complex unstructured text processing, conversational interfaces. Many have it because LLMs are the technology they've been thinking about, and so every problem looks like an LLM problem.

The cost of this isn't just over-engineering. LLMs come with real downsides — latency, cost, unpredictability, evaluation difficulty — that get hand-waved away when the team has decided this is the answer. Here are four patterns where we've watched teams make this mistake.

1. Structured data extraction from structured sources.

The single most common LLM misapplication we see: extracting specific fields from data that has structure. A team has invoices in PDF, or order confirmations in email HTML, or API responses, or CSV files. They need to extract specific fields. They reach for an LLM with a structured output prompt.

For one-off extraction at low volume, LLMs are fine and often the fastest path. For repeated extraction at volume, the failure modes — hallucinated fields, inconsistent formatting, occasional structural errors — become problems. A regex, a CSS selector, an XML path, or a proper PDF library will do the job 1000× cheaper and 10,000× more reliably.

Rule of thumb: if you can see a pattern in the data structure that wouldn't change across thousands of examples, write code that exploits the pattern. Save the LLM for genuinely unstructured cases.

2. Classification with plenty of labeled data.

Another common pattern: a team has a classification problem (categorize support tickets, classify products, route emails) and tens of thousands of labeled examples in their database. They reach for a few-shot LLM prompt.

This is genuinely backwards. With thousands of labeled examples, a fine-tuned classical model (XGBoost, a fine-tuned small language model, even logistic regression with reasonable features) will outperform a few-shot LLM on accuracy, run in milliseconds instead of seconds, and cost essentially zero to operate.

LLMs shine when you don't have labeled data — when "explain the rule in English" is faster than collecting and labeling thousands of examples. If you already have the labels, use them.

LLMs are at their best when writing prose is faster than writing code. When neither writing nor labeling is faster, you have a different kind of problem.

3. Deterministic logic dressed as AI.

The third pattern is more subtle. A team is building a workflow — maybe approving expense reports, routing customer requests, or processing orders. The workflow has rules. Some of the rules are complex. The team wraps the whole thing in an LLM agent because "agents are good at workflow."

The result is invariably worse than the alternative. The LLM occasionally violates the rules. The behavior is harder to debug. The latency is higher. The cost is higher. Compliance teams hate it.

A finite-state machine or rules engine handles deterministic workflows with deterministic behavior. Reserve LLM intelligence for the moments where genuine reasoning is needed — usually a small percentage of the overall workflow.

4. Search and retrieval where BM25 would do.

"Semantic search" has become a default answer for any search use case. The pitch is compelling: embed queries and documents, find similar ones, return relevant results. Real semantic understanding!

For some use cases this is genuinely better. For many, especially product catalog search and document search where users use specific terms, classical lexical search (BM25, Elasticsearch's default) outperforms semantic search on the metrics users actually care about.

The honest answer is usually hybrid: BM25 for lexical relevance, plus semantic search for cases where the user's terminology doesn't match the document's. But starting with embeddings and adding lexical as a fallback is the wrong direction. Start with lexical, add semantic where the lexical falls short.

The pattern behind all four.

The common pattern across these mistakes is treating LLMs as the answer rather than as one tool in a larger toolkit. LLMs are extraordinarily capable at a specific set of tasks: generating language, reasoning over context, handling genuinely unstructured input. They're mediocre or actively bad at tasks that don't require those specific capabilities.

How we make the call.

When a client proposes an LLM-based solution, we walk through three questions:

  1. What's the deterministic baseline? What would the classical (or simpler) solution look like, and how would it perform?
  2. What does the LLM add? Specifically what capability does it provide that the deterministic baseline doesn't?
  3. What does the LLM cost? Latency, dollars per request, evaluation complexity, operational complexity.

If the LLM doesn't clearly add capability that justifies its cost, we use the deterministic baseline. Most of the time, this is the right answer.
