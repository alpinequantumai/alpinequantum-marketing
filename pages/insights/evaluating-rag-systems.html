Evaluating RAG systems: what we use in practice.

Our standard evaluation framework for retrieval-augmented generation systems, with the specific metrics we report to clients and how we calibrate them.

KK
Ketul Kumar
Co-Founder & CTO · AlpineQora
April 7, 2026 13 min read

Every enterprise RAG project has the same untold problem: nobody knows whether it's working. Anecdotal feedback is positive when the demo goes well, negative when it doesn't, and most of the time genuinely unclear. Without a real evaluation framework, RAG systems either ship in a fog of guesswork or never ship at all.

AlpineQora has shipped a dozen production RAG systems across finance, healthcare, retail, and software. This is the evaluation framework that's evolved out of those engagements.

Why standard LLM evals don't work for RAG.

Off-the-shelf LLM evals (HELM, MMLU, etc.) measure capabilities of the base model. RAG system evaluation needs to measure something different: how well the system answers questions over a specific corpus. The base model is one input; the retrieval is another; the prompt structure is a third. You need an eval that captures the system end-to-end.

The other failure mode: teams build an eval framework but only measure end-to-end answer quality. This makes it impossible to debug. When the answer is bad, you don't know whether the retrieval failed (wrong context) or the generation failed (right context, wrong synthesis).

Four axes every RAG system needs to measure.

Our standard framework measures four things separately:

1. Retrieval quality

Given a query, did the retriever find the relevant documents? This is measured against a labeled set of (query, relevant-document-IDs) pairs. Metrics: precision@k, recall@k, MRR (mean reciprocal rank).

Critical detail: the labeling needs to be done by domain experts, not crowdsourced. "Relevant" is a domain-specific judgment in most enterprise corpora.

2. Context utilization

Given the right context, did the model use it correctly? We measure this by comparing answers generated with high-relevance context vs. answers generated with low-relevance context, controlling for everything else. A well-functioning system should show meaningful answer-quality difference; if it doesn't, the model isn't actually using the retrieved context.

3. Answer correctness

Given the full retrieval + generation pipeline, is the answer correct? Measured against a labeled set of (query, correct-answer) pairs. Scored by an LLM judge for fuzzy matching, with human spot-checks for calibration.

4. Answer faithfulness

Even when the answer is correct, is it grounded in the retrieved context, or is the model hallucinating? Measured by checking whether claims in the answer are supported by the retrieved documents. Critical for compliance-sensitive use cases.

An answer that's correct but unfaithful is fine in some domains and catastrophic in others. Measure both, separately.

The eval dataset: how to build one.

The hard part of RAG evaluation isn't the metrics — it's the dataset. Three principles that have served us well:

Start with real queries, not synthetic ones

LLM-generated test queries (which are easy to produce) typically don't match the distribution of real user queries. Real users ask shorter, more elliptical, more ambiguous questions. Start by collecting real queries from initial users (or beta testers, or yourself acting as a user).

Stratify by query type

RAG systems perform very differently on different query types. Lookup queries ("what's the policy on X?") are easy. Synthesis queries ("compare our policy with regulation Y") are hard. Multi-hop queries ("what's the trend in X over the past three years?") are harder still.

Build your eval set with explicit stratification across query types, and report metrics per stratum. Aggregate metrics hide important failure modes.

Grow the dataset from production failures

Every time the production system answers something incorrectly, add that query to the eval set with the correct answer. This naturally targets eval expansion at the system's weakest points. The eval set becomes a regression test as much as a benchmark.

Specific metrics we report.

For client engagements, we typically report:

  • Retrieval recall@5: are the relevant documents in the top 5 results?
  • Answer correctness rate: % of queries with correct answers, per query-type stratum
  • Answer faithfulness rate: % of answers grounded in retrieved context
  • Hallucination rate: % of answers containing claims not supported by context
  • Refusal accuracy: when the system should say "I don't know," does it?
  • Latency P50/P95: speed of end-to-end response
  • Cost per query: $ cost including embeddings, retrieval, and generation

These metrics are tracked over time, on a fixed eval set, so we can see whether the system is improving with our changes.

Tools we use.

For evaluation tooling, we typically reach for:

  • LangSmith for end-to-end trace logging and eval orchestration
  • Helicone when we need lower-cost trace logging
  • Ragas for the off-the-shelf RAG-specific metrics (faithfulness, answer relevancy)
  • Custom Python harness for the eval-set management and scheduled runs
  • OpenAI/Anthropic models as LLM judges for fuzzy answer scoring, calibrated against human judgment on a sample

Most clients end up with a mix — typically LangSmith for trace logging plus a custom Python harness for the specific eval-set discipline.

Common failure modes this framework catches.

Things this framework consistently identifies that anecdotal feedback would miss:

  • Embedding model mismatches. The system retrieves syntactically similar but semantically irrelevant content. Recall@k drops sharply.
  • Chunk-size mistakes. Chunks too small lose context; chunks too large dilute the relevant information. Both manifest as degraded answer correctness with adequate retrieval recall.
  • Prompt regressions. A "minor" prompt tweak silently degrades faithfulness or correctness. Periodic eval runs catch this.
  • Model upgrade surprises. Moving from one model to another (e.g., GPT-4 to Claude) sometimes improves overall metrics while regressing on specific query strata. Stratified metrics catch this.

What's worth the investment.

RAG evaluation is genuinely effortful. A proper eval framework typically costs us 2-3 weeks of focused work at the start of a project. It's tempting to skip it, especially when there's pressure to demo something quickly.

Almost every project where the team skipped a real eval framework ended up rebuilding it later — at higher cost and with less data than if they'd built it from the beginning. The eval framework is the foundation that everything else is built on. Treat it that way.
