Building the agent framework behind our client deployments.

Internal R&D effort: a deterministic-first agent framework that became the foundation for half a dozen client deployments. Started as a focused 8-week internal sprint, now an open-source project actively maintained by our team.

The challenge

Most agent frameworks fail in production.

By late 2025, AlpineQora had shipped four agentic systems for clients. Across those engagements, a frustrating pattern had emerged. The popular open-source agent frameworks — useful for prototypes — kept hitting the same walls when pushed into production: tool definitions buried in prompts, no proper RBAC, no audit trail suitable for compliance review, fragile error handling, expensive over-reliance on LLM calls for routing.

Every client engagement was solving the same set of problems in slightly different ways. The team was reinventing the same infrastructure components for each project. We were losing time.

We took an internal eight-week sprint to consolidate the lessons learned across those four engagements into a reusable foundation. The brief: a framework opinionated about production concerns (governance, observability, deterministic routing, tool registries) without being opinionated about specific LLMs, surfaces, or domains.

The approach

How we did it.

An R&D project but run with the discipline of a client engagement. Same sprint cadence, same code review standards, same demos.

Phase 1: Architectural decisions (weeks 1–2)

The first two weeks were architecture. The team debated and decided on the core abstractions: a tool registry (independent of agents), a deterministic intent router (handles the easy 60-70% of intents without LLM calls), a Concierge LLM that delegates to specialist agents, and an observability layer that captures every decision for audit.

Crucially, we decided against building our own LLM abstraction. The framework would adopt the existing LiteLLM-style abstractions rather than reinventing them.

Phase 2: Core framework (weeks 3–5)

Three weeks of focused implementation: tool registry with typed schemas and RBAC, deterministic router with FSM and regex matching, agent orchestration layer, observability hooks, and the contract for swapping LLM providers without changing application code.

The codebase was kept deliberately small. Every feature had to justify its weight against the alternative of just writing application code without a framework. About 4,000 lines of TypeScript at the end of phase 2.

Phase 3: Reference implementations (weeks 6–7)

To validate the design, we built three reference applications: a customer support agent, an internal HR helpdesk, and a multi-surface candidate hiring agent. Each pushed on different parts of the framework — the customer support agent stressed the routing system, the HR helpdesk stressed RBAC, the hiring agent stressed multi-surface (web + mobile + email) orchestration.

Phase 4: Open source release (week 8 and beyond)

The framework was released as open source under Apache 2.0 license. Documentation, tutorials, and example deployments. The team continues to maintain it as an active project — bug fixes, performance improvements, and community contributions.

The outcome

What we have now.

The internal effort delivered three durable outcomes:

Faster client engagements. Projects that previously required 8-12 weeks to reach a production-grade agent now ship in 4-6 weeks. The standard infrastructure is no longer being reinvented for every client.

An open-source artifact that builds credibility. The framework gets meaningful community engagement — issues, pull requests, deployments at companies we've never worked with directly. It's become a useful artifact in sales conversations, demonstrating our point of view on agentic systems.

A clearer point of view internally. The framework is opinionated about how production agents should be built. Defending those opinions across reference implementations and external use has sharpened the team's thinking. It's now a core part of our engagement methodology.
