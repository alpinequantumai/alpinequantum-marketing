Client Story · Healthcare Data Platform · 8 weeks · 3 workstreams

Cutting cloud spend by a third while doubling deployment frequency.

A large healthcare data platform was struggling with both runaway AWS costs and slow release cycles. An 8-week engagement covering Kubernetes consolidation, reserved-capacity rightsizing, and CI/CD overhaul delivered measurable wins on both axes in a single quarter.

Lumen Co.
Industry

8 weeks · 3 workstreams
The challenge

The classic cloud paradox: too expensive and too slow.

Lumen Co. provides a clinical data analytics platform used by 180+ healthcare organizations. The platform had grown rapidly over five years, with cloud costs growing even faster — AWS bills had crossed $310,000 per month and were trending sharply upward. At the same time, the engineering team was deploying less frequently, not more, as the infrastructure complexity grew.

The CFO wanted cost reduction. The CTO wanted faster releases. Both sides had been treating these as competing priorities — the assumption being that more reliable, faster deployment infrastructure would cost more. We started by challenging that assumption.

The reality was that Lumen's cloud spend was bloated because the deployment infrastructure was bad. Engineers, lacking confidence in deployments, over-provisioned generously to compensate. Slow CI/CD pushed teams to merge larger changes less frequently, which encouraged "just in case" capacity buffers. The two problems were the same problem.

The approach

How we tackled it.

We split the engagement into three parallel workstreams, each with a clear measurable outcome.

Workstream 1: Kubernetes consolidation (weeks 1–4)

Lumen was running 11 EKS clusters across two AWS regions. Many ran at 15-30% average utilization, with capacity buffers sized for worst-case load that almost never occurred. We consolidated workloads onto three production clusters and one staging cluster, with Karpenter handling dynamic node provisioning.

Workload-specific tuning — right-sized requests, proper limits, HPA thresholds — recovered substantial unused capacity. Cluster utilization went from ~22% average to ~68% average within the first month.

Workstream 2: Reserved capacity & instance optimization (weeks 2–5)

Lumen had been running entirely on on-demand instances, despite having predictable baseline load. We modeled actual usage patterns over 12 months and purchased savings plans for the steady-state base, leaving on-demand capacity only for burst load and uncertain growth.

Concurrently, we audited instance type choices. Several workloads were running on memory-optimized instances when they were actually CPU-bound; others were on general-purpose instances when memory-optimized would have been cheaper. Switching workloads to right-fit instance families recovered significant additional savings.

Workstream 3: CI/CD overhaul (weeks 3–8)

The original CI/CD setup was a tangle of Jenkins jobs, Ansible playbooks, and shell scripts accumulated over five years. We replaced it with a GitHub Actions + ArgoCD pipeline that automated everything from PR to production. Builds that had taken 32 minutes now took 6. Deployments that had been a quarterly fire drill became a routine action.

The CI/CD work also enabled the cost savings. Engineers became willing to right-size capacity once they trusted that scaling up was a fast, safe operation. The cultural shift was as important as the technical one.

The outcome

What Lumen has now.

The cost savings are recurring — that's roughly $1.4M annualized at the new run rate, against an engagement cost that paid back in under three months.

A FinOps practice the team owns. We left Lumen with a tagging strategy, a cost monitoring dashboard, and weekly review cadence that the platform team continues to run. Six months on, costs have stayed stable as the platform has grown.

A modern CI/CD platform. The GitHub Actions + ArgoCD setup is now the standard for all new services. New service onboarding takes hours instead of days.

Deployment culture shift. The team deploys multiple times per day per service, with rollbacks measured in minutes. The cultural shift compounds — engineers ship smaller, safer changes more often.
